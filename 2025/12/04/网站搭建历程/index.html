网站搭建历程

1.0 原型构建与技术选择

按照前期的调查＆请教，我选定了静态站点生成器 Hexo + GitHub Pages 的技术路线。因为这个方案无需服务器成本、对于“专业”“硬核”编程知识的依赖相对少、能用 Markdown 快速积累内容。但作为完全的小白，我发现真正的第一关不是选型，而是把本地环境和基础工具先弄通。正所谓万事开头难，我一开始就卡在了Git与 Node 的安装上：

我在Windows上反复安装、几次卸载Git 和 Node，出现过 Git Bash 无法打开、node 在 bash 中报 command not found、以及在安装包点击安装时弹出异常界面等情况。（欲哭无泪：为什么和教程说的不一样啊！）

在G老师的帮助下，我进行了一步步排查：在 Git Bash 中用正确的路径切换盘符（cd /d/hexo-blog/blog），确认 node -v 和 npm -v 输出（后来显示 v22.20.0 和 10.9.3），然后重新安装 Hexo CLI、初始化项目。

Notes——Windows 下 Git Bash 与 CMD/PowerShell 的路径写法不同，以及安装后一定要在终端里验证 node 与 npm 是否可用。

2.0 初次搭建

基础环境确认后，我在本地用 npx hexo init／hexo new 等命令搭起了一个最初的站点并在本地预览。
接着按照b站上的教程把项目推送到 GitHub：注册 GitHub 账号、在 GitHub 上创建仓库。我在初次尝试用 HTTPS 部署时多次遇到网络阻断（Recv failure: Connection was reset）和身份配置错误（deploy 时 Git 提示要配置 user.name、user.email）。为了稳定上传，我生成了 SSH key 并把公钥添加到 GitHub（ssh-keygen -t ed25519、cat ~/.ssh/id_ed25519.pub、在 GitHub → Settings → SSH keys 粘贴），然后把 _config.yml 的 repo 换成 SSH 地址 git@github.com:用户名/仓库.git。这一步虽然看起来多余，但解决了 HTTPS 时的访问受阻并让 hexo d 可以顺利把生成文件推上 GitHub Pages。

3.0 安装NexT主题

• 安装 NexT 主题，并进行基础设置
• 建立博客最初的页面结构（Home / About / Projects）
• 了解 Hexo 渲染机制、静态资源处理规则

有了部署通道之后，网站搭建重心转到了外观与内容。我先试着换主题。浏览了数十个 Hexo 主题，我最终选择了安装NexT，并初步规划网站结构为“首页”“About”“Projects ”“Blog”。
更换主题的过程中也遇到几次意外：因为国内外网络与仓库分支的问题，git clone 一度因为连接重置失败，后来在换用可用的 clone 命令并在科学网络下重试才成功；有次我误选了主题分支名称（v8.18.2），导致 fatal: Remote branch ... not found 的报错，然后改为直接 clone 默认分支就行了。成功把主题拉下来后，页面在本地渲染正常，但线上经常看不出变化。

Notes——每次改完都要hexo clean && hexo g && hexo d 来强制清理缓存、重新生成并推送。

4.0 前段样式调整——从“默认模板”到“个人风格入口”

  • 改写主题配置文件 _config.yml
  • 尝试更换头像、定制菜单、启用主题特性（社交图标、搜索等）
  • 处理头像无法显示的问题
  • 修复菜单中文件路径不生效的问题
    学习重点：前端资源路径、主题层级、渲染规则

在往头像、背景、社交图标里塞资源时，我走了很多弯路。第一次上传头像时文件名与路径问题导致头像不显示，页面只留了一个空白框；有时候图片在本地能看见，但部署后丢失。
我发现问题常常出在资源路径（主题默认用绝对/相对路径不同）、图片名大小写、以及沒有把图片放到 source/images 或 themes/.../source 的正确位置。

Notes——解决方法是把头像和 banner 统一放入 source/images/，在主题配置或页面里引用 /images/avatar.png 或 /images/banner.jpg，然后再次运行生成与部署；同时注意 Git 提示的 LF/CRLF 警告，虽然不是功能性错误，但会影响文件在不同系统上的换行显示。

我在这个环节（惨痛）学习到：静态站点的资源路径必须非常谨慎地处理，尤其在 Windows 环境下。

终于拥有了头像，顶部卡片也从默认NexT改成了我的名字

5.0 SSH 部署与 Git 配置

另外一个较大的技术点是 SSH 部署与 Git 配置：我曾收到 Hexo 部署时报 Author identity unknown 并要求运行 git config --global user.email / user.name，还有 Error: Spawn failed 的错误。我重新按提示配置了 Git 的用户信息，并清理 .deploy_git 目录（rm -rf .deploy_git），重新 hexo g && hexo d，最终成功把本地生成的 public 内容推送到 GitHub。在这一过程中，我发现生成 SSH key是不可避免的。
（试图偷懒跳过SSH密匙的计划破产）

测试ssh -T git@github.com的意义在于SSH 可以避免 HTTPS 在网络中断或凭证交互时发生问题，尤其是在频繁自动化部署的场景下更稳定可靠。

6.0 post发布

发布文章、插入图片、调整分类这些日常操作开始逐步熟悉。我学会了 hexo new post "标题" 生成草稿、在 source/_posts/ 里写文章、用 Markdown 嵌入图片（把图片放 source/images，在文章中用 ![](/images/xxx.jpg) 引用），然后用 hexo g && hexo d 发布。
通过几次实践，我从不会插图、不会推送，变成能熟练发一篇带图片的 post，并删除 Hexo 自带的 hello-world 示例文章。这个过程看似基础，但对我来说是非常重要的上手里程碑。

7.0 首页重构——最大瓶颈

  • 发现 Hexo 默认将首页设为文章列表（archive）
  • 重构导航栏、优化子页面布局
  • 尝试通过配置禁用首页文章渲染失败
  • 连续 debug，清缓存、重部署，但结果不变
  • 理解主题层覆盖与渲染逻辑限制

我想让首页是一个视觉冲击的 landing（大背景 + 居中卡片 + 介绍与社交链接），但 NexT 主题默认把“最新文章列表 (Archive)”当作首页。即便我在 _config.yml 里禁用首页文章、或把 index_generator 改成 per_page: 0，主题仍然会生成自己的 index.html，把文章列表显示在首页。
不管我怎么修改配置，打开网站永远是一排排的文章列表，那段时间我陷入了僵局。
为了解决首页之争，我做了各种尝试：修改主题配置、在 source/index.md 写自定义页面、把自定义页面改名为 home.md 并访问 /home.html 来调试。但最终突破是在更深入地理解 Hexo 与主题渲染逻辑后，采用了手写首页CSS 的方式（实现毛玻璃效果的卡片和全屏背景），编写自定义 HTML覆盖主题层（头部声明layout: false、写独立的 index.swig、用重定向使 / 指向 /home.html），并通过修改 index_generator，把原本霸占首页的文章列表赶到 /blog/ 目录下，最终才把 landing page 放在首页。
现在，访客一进来看到的是我的设计，点击 “Enter Blog” 才会进入阅读区。

8.0 新导航体系——设计音乐专题区

在音乐专题与功能分区方面，我把 Music 做成了一个有内容承载的版块，不再是散列的标签。
它包含

  • 链接到我的厦大广播电台主页（外链）
  • 把《自由音乐制作指南》做成 PDF 并嵌入网页在线预览（iframe）
  • 一个 Bookmarks 页面汇总常用乐评网站
  • 为个别专题建立独立页面并放入封面和资料

在第一版本中，这四块内容在同一页面展示，但我想要分成四个子页面，这样就可以持续更新新post。
这一步的突破点是要将原本的post改为blog并加入到menu中。Music版块的设计是我初次开始认真构思网站内容，也让我学会了如何在静态站点里同时混合外链、内嵌文档与本地页面。

9.0 细节优化

这一次更新中，我改良了许多细节：

  • 更换favicon：将浏览器标签页的小图标换成了我的个人头像（小熊）
  • 更新页脚：移除了通用的模板版权信息，改为 © 2025 Qian Miao
  • 对齐导航文字：修复了中英文混排的问题，统一了菜单的大小写
  • 图标对齐：通过修改语言包和配置文件，修复了自定义菜单的缩进 bug

我还修改了网站字体、调整字体颜色、增加滚动条，进一步统一页面视觉。

10.0 更新内容

我修改了先前的几条post，将我在搭建网站过程中的学习笔记同步进去，包括我整理的常用bash命令与markdown语法。到这里，网站已经从“能跑起来”变成“能展示我的内容和工作流程”。

从最开始连 Git Bash 都打不开，到现在可以写文章、上传图片、嵌入 PDF，把页面部署到 GitHub Pages，整个过程通过不断将抽象需求落地的实践，引导具体工具链的学习。对我而言，这种“需求导向”的学习方式比从头看网课要高效得多。
